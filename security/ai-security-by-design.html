01 🏗️
Secure Architecture Principles
Foundation-layer controls that every AI system must implement before anything else is built on top.
72%of security leaders report record-high AI risk levels in 2025, driven by insecure AI architectures deployed without ZTA principles (SANS AI Guidelines 2025)
  • Zero Trust Architecture (ZTA) — Never trust, always verify; apply to every AI pipeline hop including inference endpoints, training environments, vector stores, and model registries
  • Least Privilege Access — Models, agents, pipelines, and APIs access only what they strictly need; implement time-bound, task-scoped JIT permissions
  • Defense in Depth — Layer WAF, API gateway, prompt guardrails, model-level output filters, post-output validators, and behavioral monitoring
  • Immutable Infrastructure — AI workloads run in hardened, reproducible, signed container images; no SSH to production training nodes; use GitOps for all changes
  • Microsegmentation — Network-isolate training, fine-tuning, inference, embedding generation, and vector database services into discrete security zones
  • Secure-by-Default Configurations — All AI APIs and model endpoints ship with all capabilities disabled; enable explicitly with documented justification
  • Blast Radius Minimization — Design system boundaries so a compromised model, agent, or pipeline cannot pivot to unrelated systems or data stores
  • AI System Inventory — Maintain a continuously updated inventory of all AI models, agents, datasets, prompts, plugins, and third-party API dependencies

Data Security & Privacy
AI data is both the fuel and the primary liability — protect it across the entire training-to-inference lifecycle.
11%of data employees paste into GenAI tools is confidential — Samsung banned ChatGPT after engineers leaked proprietary source code (Cyberhaven Research 2024)
  • Data Classification & Tagging — Label all training and inference data by sensitivity (PII, PHI, CUI, FOUO, IP) before any ML pipeline ingestion; enforce classification in metadata
  • Data Minimization — Collect, retain, and process only what is strictly necessary for the AI task; enforce TTL policies on all training datasets
  • Differential Privacy (DP) — Apply calibrated Laplace or Gaussian noise to training data and statistical query results to prevent membership inference and re-identification attacks
  • Federated Learning — Train models on distributed data without centralizing raw sensitive datasets; use secure aggregation protocols to protect gradient updates
  • Encryption at Rest & In Transit — AES-256-GCM for stored datasets, model weights, and embedding vectors; mutual TLS 1.3 for all data flows between AI pipeline components
  • Secure Data Pipelines — Validate, sanitize, hash-verify, and audit all data entering AI training; reject malformed, out-of-distribution, or poisoned inputs at ingestion gates
  • Machine Unlearning / Right to Erasure — Implement selective unlearning capability to comply with GDPR Article 17; test that erased data cannot be reconstructed from model outputs
  • Data Lineage & Provenance Tracking — Log the full origin, transformation, and usage chain of every dataset used in model training; cryptographically sign each step
  • GenAI Output DLP — Deploy Data Loss Prevention controls integrated with GenAI platforms to intercept confidential data before it is submitted to external AI APIs

Model Security
Harden the model artifact itself — from training through registry to serving — against tampering, theft, and adversarial manipulation.
13%of organizations suffered breaches directly tied to their AI models or applications in 2025, with model evasion accounting for 21% of AI attack vectors (IBM CODB 2025)
  • Model Signing & Integrity Verification — Cryptographically sign all model artifacts with Sigstore/cosign; verify signatures before every deployment, fine-tuning run, and inference startup
  • Adversarial Robustness Testing — Test models against evasion (FGSM, PGD), poisoning, model inversion, and membership inference attacks using ART, Foolbox, or CleverHans before every production release
  • Backdoor Detection — Scan all models (especially downloaded from HuggingFace) for neural backdoors using Neural Cleanse, STRIP, or ActivationClustering before use in production
  • Model Watermarking — Embed traceable cryptographic or statistical identifiers in model weights to detect theft, unauthorized redistribution, or derivative use
  • Secure Model Registry — Version-controlled, RBAC-enforced model store (MLflow, DVC, HuggingFace private) with full audit history, rollback capability, and deployment approval gates
  • Continuous Red Teaming — Continuously probe models with PyRIT, Garak, and Promptfoo as mandatory CI/CD gates; block any model update that introduces new critical vulnerabilities
  • Explainability for Security Forensics (XAI) — Generate SHAP/LIME explanations for model decisions; detect anomalous decision boundaries that may indicate successful poisoning attacks
  • Anti-Extraction Controls — Implement output perturbation, query rate limits, and extraction-pattern detection to prevent model stealing via systematic API queries
  • Secure Fine-tuning Pipelines — Verify dataset provenance and integrity before every fine-tuning run; monitor output distribution shift as an indicator of data poisoning

LLM-Specific Security · OWASP Top 10
Mitigations for the 10 vulnerability classes specific to Large Language Model applications — the #1 AI security priority in 2026.
#1Prompt injection has been the top LLM vulnerability for two consecutive years per OWASP; NIST reports a 2,000%+ increase in AI-specific CVEs since 2022
  • Prompt Injection Defense (LLM01) — Sanitize all inputs; enforce instruction hierarchy (system > developer > user); use structured output schemas; deploy spotlighting defenses; never trust indirect content as instructions
  • Sensitive Info Disclosure Prevention (LLM02) — PII detection on all outputs; deny-by-default on training data retrieval; output redaction pipelines; zero-trust to sensitive data stores from inference layer
  • Supply Chain Hardening (LLM03) — SBOM for all model and library dependencies; cryptographic verification of base models; ModelScan all downloads; signed attestations for fine-tuning pipelines
  • Data & Model Poisoning Detection (LLM04) — Statistical anomaly detection on training batches; label distribution drift monitoring; hash-verify all dataset versions; isolate fine-tuning from user-supplied data
  • Improper Output Handling (LLM05) — Treat LLM output as untrusted input; apply HTML/SQL encoding before rendering; never execute raw LLM code without sandbox; validate JSON schema of structured outputs
  • Excessive Agency Controls (LLM06) — Define explicit tool allowlists per agent role; require HITL approval for irreversible actions; scope permissions per task; implement maximum action budgets
  • System Prompt Leakage Prevention (LLM07) — Never store secrets in system prompts; test prompt confidentiality with adversarial extraction probes; use separate secure secret management
  • Vector & Embedding Security (LLM08) — RBAC on vector databases; validate retrieved chunks before RAG injection; monitor cosine similarity distributions for poisoning signals; audit embedding access logs
  • Misinformation Mitigation (LLM09) — Factual grounding via authoritative RAG sources; confidence scoring; citation verification; mandatory disclaimers for low-certainty outputs; domain expert review gates
  • Unbounded Consumption Controls (LLM10) — Hard token and compute budgets per user/session; adaptive rate limiting; detect context-stuffing attacks; cost anomaly alerting; circuit breakers on API runaway

Identity & Access Management
In 2026, identity is the #1 attack target — with AI agents outnumbering human users 82:1, non-human identity management is now critical.
300K+ChatGPT credential sets were advertised on the dark web in 2025 via infostealer malware — AI service credentials are now high-value targets (IBM X-Force 2026)
  • AI Workload Identity (SPIFFE/SPIRE) — Issue cryptographic SVID identities to every model, agent, pipeline stage, and serving endpoint; eliminate long-lived secrets in all container environments
  • Phishing-Resistant MFA — Enforce FIDO2/Passkey authentication for all human access to AI platforms, model registries, and training environments; ban SMS-based MFA for AI system access
  • Fine-Grained RBAC/ABAC for AI — Control access to specific models, datasets, embedding stores, and inference APIs based on roles AND attributes; separate permissions for dev/staging/prod environments
  • API Key Vault Integration — Store all AI service credentials (OpenAI, Anthropic, HuggingFace, Cohere) in HashiCorp Vault or AWS Secrets Manager; automated 30-day rotation; never hardcode in notebooks
  • Service Mesh mTLS Authentication — Enforce bidirectional certificate-based authentication via Istio or Linkerd between all AI microservices; no plaintext inter-service AI communication
  • Non-Human Identity (NHI) Lifecycle — Apply full IAM lifecycle controls (onboard, rotate, offboard) to every AI agent credential; monitor for anomalous agent behavior as early compromise indicator
  • Just-in-Time (JIT) Access — Grant temporary, task-scoped access for model training, fine-tuning, and data access jobs; automatically revoke upon completion via privileged access workstations
  • Credential Exposure Monitoring — Continuously scan dark web and public repositories for leaked AI service API keys; use Trufflehog or GitGuardian in pre-commit hooks

Multi-Agent & Agentic System Security
The fastest-growing attack surface in 2026 — AI agents with real-world tool access require their own dedicated security architecture.
82:1AI agents already outnumber human users in enterprise environments; OWASP published a separate Top 10 for Agentic Applications in late 2025 confirming this requires its own framework
  • Explicit Agent Permission Scoping — Define tool and resource allowlists per agent role using deny-by-default; never give agents broader permissions than the minimum required for the current task
  • Code Execution Sandboxing — Run all agent-generated code in gVisor, Firecracker, or WASM sandboxes; restrict filesystem access, network calls, and process spawning per sandbox policy
  • Human-in-the-Loop (HITL) Gates — Require explicit human approval before irreversible agent actions: send emails, modify production databases, make purchases, deploy infrastructure, or delete data
  • Inter-Agent Message Signing — Sign and verify all messages between orchestrator agents and sub-agents using HMAC or asymmetric keys; reject unsigned or tampered inter-agent instructions
  • Blast Radius Isolation — Network-segment agent execution environments; prevent any single compromised agent from accessing unrelated systems, user data, or sibling agents
  • Full Agent Forensic Audit Trail — Log every decision, tool call, memory read/write, external API call, and credential access with causality chains; store immutably for 90+ day forensic retention
  • Indirect Prompt Injection Defense — Sanitize all external content (retrieved web pages, emails, files, database records) before injecting into agent context; never interpret external content as trusted instructions
  • Agent Identity Hierarchy & Trust Levels — Define explicit trust tiers (orchestrator, sub-agent, tool); sub-agents cannot self-escalate permissions or directly request higher-trust actions
  • Convergence & Loop Prevention — Implement hard maximum action budgets, iteration limits, and loop detection with automatic circuit-breaker suspension of runaway agents
